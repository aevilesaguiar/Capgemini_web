SSL

  • - Protocolo de Camada de Sockets Segura (secure socketes layer);
  • - Há também o TLS(Segurança da Camada de Transporte/Transporte Layer)
  • - Alguns consideram o SSL depreciado e sugerem o uso do TLS
  • - É um protocolo criptografado que permite ter mais segurança de comunicação ao acessar um ambiente virtual páginas web ou e-mail.
  • - Os dados são criptografados, impossibilitando outras pessoas de utilizar.
  • - Quando não usamos protocolo de segurança as informações são transmitidas de forma literal e pessoas mal intencionadas podem acessar os dados de cartão, cpf...
  • - Geralmente o usuário pode escolher em usar sem o protocolo de segurança ou com https
  • - A instalação é feita normalmente através do servidor contratado, pode ser de forma gratuita ou paga(acesso a suporta 24hrs)
  • - No wordpress há plugins para esta funcionalidade;

Hoje ainda é mais usado o SSL , mas já está havendo migrações para o TLS.

CORS

Cross-Origin Resource Sharing ou CORS é um mecanismo que permite que recursos restritos em uma página da web sejam recuperados por outro domínio fora do domínio ao qual pertence o recurso que será recuperado

Cors de uma forma geral é um compartilhamento de recursos de origem cruzadas em origens diferentes

é um mecanismo que usa cabeçalhos adicionais HTTP para permitir que um WebApp seja executado numa origem/dominio com permissão para acessar recursos pré selecionados de um outro servidor

Executa requisição Cross-origin HTTP ao solicitar um recurso de origem diferentespodendo ser um domínio ou protocolo ou porta diferente da origem.

Ppode ocorrer nos testes que utilizamos recursos em portas diferentes. Exemplo: http://localhost:3000/ e http://localhost:3001/ .

Ou usar recursos de outros domínios dentro do nosso WebApp, como imagens que não estão no nosso domínio, para fazer isso temos que usar um domínio de origem cruzada.

Cors: Possiveis erros

Lembrando que CORS se trata de requisições e usos de recursos de outro servidor ou portas.

Erros mais comuns são com as requisições JavaScript que utilizam XMLHttpRequest ou Fetch APIs.

  • - XMLHttpRequest são objetos usados para interagir entre servidores, indo desde coisas mais simples como obtenção/recuperação de dados dentro de uma URL até atualização completa de página.
  • - Fetch APIs são interfaces para buscar recursos, usado para acessar e manipular HTML, com requests e reposnse objects.
  • - Erros de CSS Shapes,
  • - Erros de carregamento de imagens

CORS: Algumas soluções:

Geralmente os problemas são causados por ausência de header.Temos esses tipos de inclusões que podemos fazer. Exemplos:

  • - Header set Acess-Control-Allow-Origin: https://google.com.br
  • - Header set Acess-Control-Allow-Methods: POST,GET : podemos corrigir através dos métodos
  • - Header set Acess-Control-Allow-hEADERS:*/*Podendo ser Accept para buscar soluções adicionais corrigir através do cabeçalho
  • - Header set Acess-Control-Max-Age:86400 /*tempo de manutenção*/ 86400 é um dia, o tempo de obtenção de cache

Primeiro precisamos saber qual plataforma estamos usando, e buscar a docuemntação para resolver os erros.

CSP

O que é o OWASP Top 10? Quais são as vulnerabilidades mais comuns?

OWASP — sigla para Open Web Application Security Project ou Projeto Aberto de Segurança em Aplicações Web, em tradução livre. Trata-se, a grosso modo, de uma comunidade global fundada em setembro de 2001 por Mark Curphey.

Ela é composta por desenvolvedores, pesquisadores e especialistas em segurança da informação, que trocam conhecimentos de forma aberta e disponibilizam gratuitamente conteúdos educacionais — incluindo artigos técnicos, tutoriais, pesquisas, documentações e ferramentas de código aberto — para auxiliar profissionais da área. O objetivo final da OWASP é mitigar vulnerabilidades de segurança na web, e, por isso, a iniciativa se organiza em diferentes capítulos locais para elaborar ações de conscientização.

A principal “obra” dessa comunidade, porém, é sem dúvida alguma o ranking anual OWASP Top 10. Trata-se de uma listagem, atualizada frequentemente, que relaciona as brechas mais críticas, comuns e perigosas quando o assunto é desenvolvimento de projetos web. O projeto começou de forma descompromissada, mas, ao longo dos anos, por conta de sua qualidade e inegável precisão, se tornou um verdadeiro guia reconhecido globalmente por desenvolvedores que desejam se esquivar de ameaças mais frequentes em ambientes web.

Brechas mais Comuns:

  • 1) Injeção: falhas de injeção (incluindo SQL, OS e LDAP) ocorrem quando um interpretador pode ser manipulado por um atacante para que ele execute comandos maliciosos, culminando na invasão de sistemas e roubo de dados pessoais em um banco de dados;
  • 2) Quebra de Autenticação: problemas na implementação correta de recursos de autenticação podem ser explorados por criminosos para comprometer credenciais e tokens, obtendo acesso não autorizado a sistemas privados;
  • 3) Exposição de Dados Sensíveis: APIs mal configuradas ou servidores públicos podem acabar expondo dados pessoalmente identificáveis, algo gravíssimo especialmente em tempos de Lei Geral de Proteção de Dados (LGPD).

Vale observar que o OWASP Top 10 não se limita a listar vulnerabilidades importantes, mas também oferece uma série de dicas e truques para que você evite esse tipo de problema em seus projetos, explicando as melhores práticas para que um projeto web se torne livre desses bugs.
